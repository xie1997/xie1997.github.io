K8s中默认组件的权限

云安全
技术文章
发布日期:   2022-11-03
更新日期:   2023-03-07
文章字数:   1.7k
阅读时长:   7 分

在K8s中的默认组件如kubectl、kubelet、scheduler、calico、pod等都需要和api server进行通信，因此也设计涉及到了这些组件的权限问题。接下来，让我们看看这些组件的默认权限。

k8s中”X509 客户证书”和”服务账号令牌”应该是身份认证最常见的两种方式，都能代表”用户”或”用户组”信息。大部分的组件(包括scheduler、calico、kubectl、pod)鉴权都是依赖RBAC模型，也就是主体绑定的角色规定了能访问哪些资源。kubelet的鉴权比较特殊，是依赖Node鉴权模型。

首先对cluster的certificate-authority-data进行解码，如下，可以看到是cluster集群相关的东西，可以看到Subject的CN参数为kubernetes，代表kubernetes用户。

echo base64编码的数据 | base64 -d | openssl x509  -noout -text

然后对user的client-certificate-data进行解码，可以看到subject是O=system:masters, CN=kubernetes-admin。kubernetes-admin是用户名，而system:masters是用户组。

查询kubernetes-admin用户和system:masters组的集群角色绑定，可以看到system:masters组的集群角色绑定是cluster-admin。

#查看集群角色绑定
kubectl get ClusterRoleBinding -A -o wide | grep system:masters

最后查看集群角色绑定cluster-admin的rules规则可以得知该集群角色绑定对集群有最高权限。

kubectl get ClusterRole/cluster-admin -o yaml

kubelet

在每个node节点上都有kubelet进程，该进程需要与API Server进行通信，而通信过程中使用的配置文件是/etc/kubernetes/kubelet.conf。查看/etc/kubernetes/kubelet.conf文件内容，

可以看到是通过/var/lib/kubelet/pki/kubelet-client-current.pem证书进行认证的。

该证书内容如下

对其进行解码，如下，可以看到subject的用户为system:node:k8s-node1，组为system:nodes。

openssl x509 -in /var/lib/kubelet/pki/kubelet-client-current.pem -noout -text

查询system:node:k8s-node1用户和system:nodes组的集群角色绑定，可以看到system:nodes组的集群角色绑定是system:certificates.k8s.io:certificatesigningrequests:selfnodeclient。

最后查看集群角色绑定system:certificates.k8s.io:certificatesigningrequests:selfnodeclient的rules规则可以得知该集群角色绑定只对当前node节点具有某些权限。

kubectl get ClusterRole/system:certificates.k8s.io:certificatesigningrequests:selfnodeclient -o yaml

在使用Node鉴权的文档中提到，kube apiserver对kubelet的鉴权比较特殊。当发现请求的用户在system:nodes组中，用户名是system:node:时，就限制这个请求只能做有限的操作，比如：

  • 读操作

    • node/pod/services等
    • 绑定到当前node的pod的secret、configMap

  • 写操作（如果开启了NodeRestriction准入插件，就只能修改kubelet所在node的资源）

    • 创建节点、修改节点状态
    • 创建pod、pod状态

注：如果没有开启NodeRestriction准入插件，就能让kubelet在任意node上创建pod。如果开启了NodeRestriction准入插件，就只能查询node、pod等低权限操作。

也不能创建pod

不让创建pod的原因主要由于是高版本的Kubernetes的准入控制器的NodeRestriction机制。高版本K8s的认证模式由之前的RBAC更改为Node,RBAC，增加了Node模式，并且
默认情况下Kubernetes禁用了外部直接对pod有写权限的操作。

如果要强制在外部创建pod，可以把/etc/kubernetes/manifests/kube-apiserver.yaml中的

- --enable-admission-plugins=NodeRestriction 
改成 
- --enable-admission-plugins=ServiceAccount

scheduler

kube-scheduler进程只在master节点运行，默认使用/etc/kubernetes/scheduler.conf配置文件。如下图查看该文件

我们对user下的client-certificate-data数据进行解码。可以看到对应的用户为CN=system:kube-scheduler。

查看该用户对应的集群角色，可以看到绑定到两个集群角色：system:kube-scheduler、system:volume-scheduler。

kubectl get ClusterRoleBindings -o wide |grep system:kube-scheduler

查看这两个集群角色的权限，可以看到只有部分权限。

kubectl get ClusterRole/system:kube-scheduler -o yaml
kubectl get ClusterRole/system:volume-scheduler -o yaml

calico

calico进程会在每个node和master节点都运行，默认使用/etc/cni/net.d/calico-kubeconfig配置文件。

我们在node1查看该配置文件内容，可以看到User下是使用的token令牌。

这个token令牌是一个jwt字符串，我们可以在在线网站https://jwt.io/#debugger对其进行解码，可以看到serviceaccount是calico-node，命名空间是kube-system，pod名字为calico-node-mnvln。

也就是对应的这个pod。

可以看到该serviceaccount绑定的集群角色为calico-node。

最后查看集群角色绑定calico-node的rules规则可以得知权限。

kubectl get ClusterRole/calico-node -o yaml

pod

一个 pod 与一个服务账户相关联，该服务账户的Token凭证被放入该pod中每个容器的文件/var/run/secrets/kubernetes.io/serviceaccount/token内。该文件的内容也是jwt格式的，我们可以在在线网站https://jwt.io/#debugger对其进行解码，可以看到serviceaccount是default，命名空间是test，pod名字为nginx-deploy4-6799db7594-84rqv。

我们来查看该serviceaccount绑定的role，发现test:default ServiceAccount没有绑定到角色。

kubectl get RoleBindings -n test -o wide | grep default
kubectl get ClusterRoleBindings -o wide  | grep test:default

而ServiceAccount账号默认会加入到system:serviceaccounts组中，我们查看system:serviceaccounts组绑定的角色可以看到其绑定了名为system:service-account-issuer-discovery的ClusterRole。

kubectl get ClusterRoleBindings -o wide  | grep system:serviceaccounts
kubectl get RoleBindings -n test -o wide | grep system:serviceaccounts

查看该ClusterRole/system:service-account-issuer-discovery的信息，可以看到该角色具有的权限很低，连查询node、pod等权限都没有。

kubectl get ClusterRole/system:service-account-issuer-discovery -o yaml

其实当我们拿到了node节点的权限，可以通过过滤token来查看运行在该node节点上所有pod的token。

在宿主机全局查找token文件，就可以看到在宿主机挂载的token文件。
